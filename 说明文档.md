# AI字体生产工具 - 说明文档

## 概述

AI字体生产工具是一套基于 Flask 的 Web 应用（端口 7550），用于 AI 字体模型的训练、字库生字和 OCR 识别验证。与高迪字库预处理工具（端口 7500）配合使用，覆盖从字体预处理到 AI 生成的完整工作流。

## 启动方式

双击 `启动.bat` 或在项目目录下运行：

```bash
python app.py
```

访问地址：`http://localhost:7550`

依赖安装：`pip install -r requirements.txt`

## 三个功能页面

### 页面一：模型训练 `/train`

基于开源项目 [zi2zi-JiT](https://github.com/kaonashi-tyc/zi2zi-JiT)，通过 LoRA 微调学习目标字体风格。

**左侧控制面板：**
- 路径配置：基础模型、源字体、学习字库 TTF、输出目录（均带路径浏览和历史记录）
- 准备数据：
  - 取字数量选择：800 字 / 3000 字 / 全部 / 手动输入
  - 准备数据按钮：从学习字库中按指定数量提取汉字，生成训练图片
  - 打开目录按钮：直接打开数据目录检查准备结果
  - 每次准备数据自动创建带时间戳的独立子目录（如 `train_20260417_143000/`），不覆盖历史数据
- 训练参数：Epochs、Batch Size、LoRA r/alpha、CFG、num_fonts、num_chars、max_chars/font、num_workers
- 操作按钮：开始训练、停止

**右侧监控区：**
- 状态指示灯（就绪/训练中/完成/错误）
- 统计面板：Epoch 进度、Loss、学习率、耗时
- 进度条
- 日志终端（黑底，实时滚动更新训练输出）

### 页面二：字库生字 `/generate`

基于训练好的模型，批量生成缺失字符的字体图片。

**四种输入模式（Tab 切换）：**

| Tab | 模式 | 说明 |
|-----|------|------|
| 0 | 文本输入 | 直接粘贴需要生成的汉字 |
| 1 | TTF + 文本 | 上传 TTF 和文本文件，计算字体中缺失的字符（支持 txt/docx/mobi/epub/pdf） |
| 2 | TTF + 字符集 | TTF 与 GB2312/GBK/Big5 字符集差集，按 500 字分组，勾选组别后逐组迭代生成 |
| 3 | 失败重试 | 粘贴迭代失败字符，指定源目录重试 |

**通用配置：** AI 模型路径、生成倍数、迭代轮次、置信阈值、CFG

**工具按钮：**
- 开始生成 / 停止
- 打开结果目录
- 去汉字后缀（`uni4E00_一.png` → `uni4E00.png`，用于 FontLab 导入）
- 加汉字后缀（`uni4E00.png` → `uni4E00_一.png`）

**注意：** 生成的图片默认带汉字后缀（如 `uni4E00_一.png`），可直接查看识别结果。导入 FontLab 前使用"去汉字后缀"即可。

**TTF+字符集分组队列：**
1. 计算 TTF 与字符集的差集，自动按 500 字/组分组
2. 每组输出独立 txt 文件：`missing_GB2312_01.txt`、`missing_GB2312_02.txt`...
3. 分组以复选框队列展示，支持全选
4. 勾选后点击"开始生成"，按勾选顺序逐组迭代
5. 每组输出到子目录：`输出目录/group_01/`、`输出目录/group_02/`...

**右侧结果区：** 进度条、成功率、卡片网格（深灰背景，莫兰迪灰卡片，支持右键删除）

### 页面三：OCR 识别 `/ocr`

基于开源项目 [PaddleOCR](https://github.com/PaddlePaddle/PaddleOCR)（当前版本 2.7.3），对生成的字体图片进行批量识别验证。

**左侧控制面板：**
- OCR 环境（独立 venv 的 Python 路径）
- 输入目录（带路径浏览和历史记录）
- 文件过滤（默认 `*.png`）
- 识别模型：中英文标准 / 服务器版 / 手写体（默认） / 繁体中文 / 英文 / 日文 / 韩文
  - 注意：当前 PaddleOCR 2.7.3 无专用手写体模型，手写体和服务器版均使用 PP-OCRv4 `ch` 模型
- 置信阈值（默认 0.5）
- 批量大小（默认 50）
- 操作按钮：开始识别、停止、导出 CSV、加汉字后缀、去汉字后缀

**右侧结果区：**
- 顶部进度条和统计（进度、已处理/总数、通过/未通过、耗时）
- 卡片网格（深灰背景，莫兰迪灰卡片）：
  - 黑色正方形图片区
  - 识别文字 + PASS/FAIL 状态标签
  - 文件名（2 倍字号，去掉 `.png` 后缀）
  - 右键单击可删除卡片及对应图片文件
  - 加/减汉字后缀后实时更新卡片显示
  - 删除后统计数据自动刷新
  - F5 刷新后自动过滤已删除文件
- 底部汇总：总数、通过、未通过、平均置信度

## 关键功能特性

### 路径历史记录

所有带 `data-persist` 属性的路径输入框会自动保存到 localStorage：
- 页面刷新后自动恢复上次使用的路径
- 浏览弹窗顶部显示历史记录，可快速跳转
- 每个输入框独立维护最多 10 条历史

### 文件/目录浏览器

所有路径输入均配备浏览弹窗，支持：
- Windows 盘符快捷切换
- 目录/文件层级导航
- 手动输入路径后回车确认
- 目录模式下"选择当前目录"确认按钮

### FontLab 命名规范

生成的图片文件名遵循 FontLab 命名规则：
- BMP 字符：`uniXXXX.png`（如 `uni4E00.png`）
- 扩展区字符：`uXXXXX.png`（如 `u20000.png`）
- 默认带汉字后缀：`uni4E00_一.png`、`u20000_𠀀.png`

### 路径回退机制

图片显示和文件删除操作支持路径回退：
- 查找带后缀文件时，自动尝试无后缀版本
- 查找无后缀文件时，自动尝试带后缀版本
- 确保加/减汉字后缀操作后页面功能不受影响

## 技术要点

| 项目 | 方案 |
|------|------|
| 框架 | Flask 3.x，Jinja2 模板 |
| 端口 | 7550 |
| 样式 | 莫兰迪色系，小圆角（按钮 6px、输入框 4px、卡片 4px） |
| 训练进度 | subprocess stdout → 日志文件 → 前端 2 秒轮询 |
| OCR 调用 | subprocess 调用独立 venv，通过临时 JSON 文件传递路径 |
| 图片懒加载 | `<img loading="lazy">` + DocumentFragment 批量渲染 |
| 字符集计算 | fontTools 读取 TTF，与 GB2312/GBK/Big5 字符集差集 |
| 文本解析 | 支持从 txt/docx（python-docx）/pdf（PyMuPDF）/epub（zip 解析）/mobi 提取文本 |
| 训练数据管理 | 每次准备数据创建独立时间戳目录，历史数据不覆盖，用户手动管理 |
| 默认配置 | config.py 集中管理，所有路径预填当前 V6 训练配置 |

## 与高迪字库预处理工具的关系

| 工具 | 端口 | 功能 |
|------|------|------|
| 高迪字库预处理工具 | 7500 | 原始图片处理：缩放校正、切割布局、标注出图 |
| AI 字体生产工具 | 7550 | AI 模型训练、字库生字、OCR 验证 |

两个工具共享相同的设计风格（莫兰迪色系、统一题头图片），通过页面导航按钮可互相跳转。

## 更新日志

### 2026-04-18 智能训练 + 继续训练 + 测试生成 + CJK 扩展区支持

#### 1. 智能训练（`smart_train.py`）

训练数据准备增加指纹缓存机制，避免重复生成数据：
- 计算目标 TTF + 源 TTF + 训练参数的 MD5 指纹，保存为 `.data_fingerprint.json`
- 指纹不变时自动跳过数据准备，直接复用已有数据
- 指纹变化时重新生成训练数据

#### 2. 继续训练

训练页面新增"继续训练 / 测试生成"控制组：
- Checkpoint 路径输入框（带文件浏览器），训练完成后自动填入 `checkpoint-last.pth`
- 继续训练按钮 + 追加轮次参数（默认 50）
- 后端自动读取 checkpoint 的 epoch，确保 `total_epochs > checkpoint_epoch`，避免训练循环为空直接退出
- 训练目录过滤跳过 `_` 前缀的目录（如 `_test_quick_output`），避免干扰 `FontSrcTargetRefsDataset` 加载

#### 3. 测试生成

训练完成后一键测试模型效果：
- 生成 20 个已有训练字符 + 20 个全新字符
- 已有字符从训练数据文件名中提取，全新字符从学习字库 cmap 中选取
- 预渲染验证：通过 cmap 检查 + 像素检测，确保只选取两个字体都能正确渲染的字符
- 输出为 target|gen 对比图片，以 OCR 卡片样式展示（已有/新字分区，带状态标签）

#### 4. CJK 扩展区完整支持

全面修复 CJK Extension B-G 字符的处理，涵盖 Python 后端和 JavaScript 前端：

**Python 端：**
- `charset_utils.py`：新增 `is_cjk()` 函数，覆盖完整 CJK 范围（Ext A-G + 兼容区），替代各处硬编码的范围判断
- `app.py`：缺失字符计算改用 `is_cjk()`
- `train_manager.py`：训练字符选取范围补充 Ext C-G
- `.ttc` 集合字体安全打开：`_open_font()` 自动指定 `fontNumber=0`

**JavaScript 端：**
- 字符计数使用 `Array.from(str).length`，正确处理 surrogate pair
- 字符拆分使用 `[...str]` 替代 `.split('')`，避免拆坏扩展区字符

#### 5. 字库生字 - 扩展区字符渲染

解决 source font 不支持 CJK 扩展区字符导致生成错误（口+X 方框）的问题：
- 渲染前通过 fontTools cmap 精确检查字符是否在字体中
- source font cmap 不包含的字符自动 fallback 到 `simsunb.ttf`（宋体扩展B，覆盖 Ext B/C/D/E 共 52876 字）
- 无法渲染的字符在生成日志中明确提示并跳过

#### 6. 字库生字 - 输出子目录 + 卡片展示

所有 Tab（文本输入、TTF+文本、失败重试）的生成结果现在与"TTF+字符集"行为一致：
- 输出在用户指定目录下创建带时间戳的子目录 `gen_YYYYMMDD_HHMMSS/`
- 生成完成后右侧自动展示卡片网格
- `get_status()` 返回 `output_dir` 字段供前端加载卡片

#### 7. 字库生字 - 进度统计修复

修复进度显示异常（如 2272/9）：
- `success_chars` 只计入本次任务字符中的成功数，不再包含历史目录中的无关文件
- 每轮迭代结束后正确更新 `success_chars`
